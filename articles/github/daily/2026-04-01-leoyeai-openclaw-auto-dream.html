これは何か

openclaw-auto-dream は、OpenClaw 系エージェントの記憶を「睡眠中の整理」に見立てて自動保守する拡張です。README では、日次ログを収集し、重要度で整理し、忘却ルールを適用しながら、ダッシュボードや通知まで返す Dream Cycle を中心機能として説明しています。

何ができるか

処理は 3 段階です。Collect で過去 7 日の未整理ログを走査し、重要マーカーや決定事項を抽出します。Consolidate で内容を 5 層メモリに振り分け、重複をまとめ、ID と関連リンクを付けます。Evaluate では重要度スコア、忘却曲線、ヘルス指標、洞察メモを生成し、最終的に dream report と通知まで作ります。

目立つポイント

特徴は 5 層メモリ設計です。作業中の圧縮記憶、エピソード、長期記憶、手順集、インデックスを分けて管理し、単なるログ蓄積で終わらせません。さらに 90 日以上参照されず重要度が低い項目は削除ではなく memory/archive.md に 1 行要約で退避し、知識グラフの接続性や鮮度まで数値で見られるのが実務的です。

セットアップや使い方の流れ

導入は README 上では単純で、clawhub install openclaw-auto-dream かエージェント経由のセットアップから始めます。その後「Set up Auto-Dream」と指示すると、タイムゾーンに合わせた毎日 4 時の cron、memory/index.json の初期化、通知レベル確認、初回 dream cycle 実行まで自動で進む想定です。必要なら「Dream now」で手動実行し、「Show memory dashboard」で可視化できます。

どんな人向けか

OpenClaw を長く使っていて、会話ログや日報が積み上がるほど記憶の質が落ちる人に向いています。特に、プロジェクトの経緯、判断理由、定型手順をあとから拾い直したい個人開発者や小規模チームとは相性が良さそうです。

注意点

README の前提は OpenClaw / MyClaw 系の運用フローに寄っているため、一般的な LLM ツールへそのまま差し込めるわけではありません。また、秘密情報を新たに検出して保護する仕組みではなく、「すでに存在する secrets を統合対象にするだけ」という Safety 方針なので、元ログ設計が雑だと記憶層にもそのまま持ち上がります。

まとめ

openclaw-auto-dream は、メモリを増やすツールというより、増え続ける記憶を腐らせないためのメンテナンス層です。ログ収集、意味的重複排除、緩やかな忘却、健康診断、ダッシュボードまでが一連でまとまっているので、OpenClaw の長期運用に欠けがちな「記憶の衛生管理」を埋める用途で見ると分かりやすいです。
