これは何か

litellm は、OpenAI、Anthropic、Azure、Vertex AI、Bedrock など多数の API を、ほぼ同じ呼び出し形で扱える Python SDK と AI Gateway を提供するプロジェクトです。

README では Python ライブラリとしての利用だけでなく、中央集約型の Proxy Server として置き、複数チームや複数アプリの LLM 呼び出しをまとめる使い方が前面に出ています。

何ができるか

チャット補完だけでなく、/responses、埋め込み、画像、音声、バッチ、rerank、A2A Agent、MCP ツール接続まで 1 つのインターフェースに寄せられます。

AI Gateway 側では virtual keys、認可、ルーティング、fallback、負荷分散、コスト追跡、ガードレール、監視 UI まで扱えるので、アプリ側の接続コードを薄くできます。

目立つポイント

MCP Gateway や A2A Agent Gateway が README の早い位置にあり、単なる API ラッパーではなく、エージェント時代の接続面を担う設計になっています。

SDK と Proxy の両輪で使えるため、個人開発ならライブラリ導入、組織運用ならゲートウェイ集約、という切り分けがしやすいです。

セットアップや使い方の流れ

最小構成なら pip install litellm で SDK を入れ、モデル名に openai/... や anthropic/... のような provider 接頭辞を付けて呼びます。

運用寄りに使う場合は pip install 'litellm[proxy]' で Proxy を立て、OpenAI SDK の base_url を LiteLLM 側へ向ける流れです。

どんな人向けか / 注意点

複数モデルを切り替えたい開発者、社内で LLM 呼び出しを標準化したい Platform / GenAI チーム、MCP や agent 接続を 1 つの面で管理したい人に向いています。

一方で設定面はかなり広く、Proxy 運用や認可設計まで踏み込むと構成は重くなります。まずは SDK か最小 Proxy から入る方が安全です。

まとめ

litellm は、LLM を増やすたびにアプリ側を作り直す負担を減らし、接続と運用をまとめるための現実的な基盤です。2026-04-04 時点の Trending Rank 3 に入っているのも、単なるラッパー以上の需要があることを示しています。
